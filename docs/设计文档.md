# 复习资料智能学习助手 — 设计文档

- **日期**：2026-06-17
- **类型**：课程设计作业（4 人 / 1~2 周 / 团队 Java·Spring 半熟）
- **目标**：做一个真能用、能演示、有 AI 亮点的最小可用平台（MVP）

---

## 1. 项目概述

一个基于 Spring AI 的复习资料智能学习平台。用户登录后上传自己的复习资料（文本/PDF/Word，以及图片、录音等多模态资料），系统将资料统一提取为文本、切片并向量化入库；用户可基于自己的资料进行问答（答案带原文出处），当自有资料无法回答时自动联网搜索补充作答；并可让 AI 根据资料自动生成练习题。

### 核心闭环

```
登录 → 上传复习资料(文本/PDF/Word/图片，录音为加分项) → 提取为文本 + 切片 + 向量化入库(按用户隔离)
     → 提问：先检索自己的资料
              ├─ 命中 → 基于资料回答(带原文出处)
              └─ 资料不足 → 联网搜索补充回答(带网页来源)
     → 选一份资料/主题 → AI 自动生成练习题(选择/填空/简答)
```

### 四个功能点

1. 多用户登录（小范围，每人独立资料库）
2. 资料导入 + RAG 问答
3. 联网问答兜底（自有资料答不上来时联网补充）
4. AI 出题

### 多模态资料（图片 / 录音）

资料来源不止文本，还可能是**图片**（拍照的笔记、教材页、图表）和**录音**（讲课音频）。处理思路：所有格式先经"提取器"统一转成文本，之后切片/向量化/检索/问答完全复用现有流程（详见第 5 节）。

- **图片**：纳入 MVP，用通义视觉模型 `qwen-vl` 识别图中文字与内容
- **录音**：列为加分项，用通义语音识别 `paraformer` 转文字稿；因文件大、识别耗时（异步），核心闭环跑通后有余力再做

### 明确不做（YAGNI，列为加分项）

- 自动批改 / 错题本
- 复杂权限、角色管理
- 高并发优化、复杂运维部署
- 搜索并导入外部资料（本期只做"联网问答"，不做"联网搜资料入库"）
- 录音转写（加分项，非核心）

## 2. 后端架构（分层）

- **Controller 层**
  - `AuthController` — 注册 / 登录
  - `MaterialController` — 上传 / 列表 / 删除
  - `ChatController` — 问答
  - `QuizController` — 出题
- **Service 层**
  - `AuthService` — 用户认证
  - `MaterialService` — 解析 → 切片 → embedding → 入库
  - `RagService` — 检索 + 问答 + 联网兜底
  - `QuizService` — 出题
- **AI 集成**：Spring AI 的 `ChatClient` + `EmbeddingModel`（通义）+ `VectorStore`
- **持久层**：MySQL 存 `users`、`materials`；向量先用文件版 `SimpleVectorStore`（零额外部署）
- **安全**：Spring Security + JWT（简单版）

### 用户隔离机制

每个文本切片入库时携带 metadata `{userId, materialId, source}`；检索时按 `userId` 过滤，保证每个用户只能检索到自己的资料。

## 3. 数据模型

### users

| 字段 | 说明 |
|---|---|
| id | 主键 |
| username | 用户名（唯一） |
| password | 加密存储（BCrypt） |
| created_at | 创建时间 |

### materials

| 字段 | 说明 |
|---|---|
| id | 主键 |
| user_id | 所属用户 |
| filename | 原始文件名 |
| type | 文件类型（txt/pdf/word/image/audio） |
| status | 处理状态（PROCESSING/READY/FAILED） |
| created_at | 创建时间 |

### 向量库 chunk

- 内容：切片文本
- metadata：`{ userId, materialId, source(文件名/页码) }` — 用于用户隔离与回答溯源

## 4. 关键数据流

### 4.1 资料上传与入库

1. 用户上传文件 → `MaterialController`
2. `MaterialService` 按文件类型分发到对应**提取器**，统一得到纯文本（详见第 5 节）
   - txt/PDF/Word → Apache Tika
   - 图片 → 通义 `qwen-vl` 视觉识别
   - 录音（加分项）→ 通义 `paraformer` 语音识别
3. 文本切片（按长度 + 重叠）
4. 每个切片调用 `EmbeddingModel` 生成向量
5. 带 metadata 写入 `VectorStore`
6. `materials` 表记录元信息与状态

### 4.2 RAG 问答（含联网兜底）

1. 用户提问 → `ChatController` → `RagService`
2. 对问题做 embedding，在 `VectorStore` 中按 `userId` 过滤检索 Top-K 切片
3. 若检索到足够相关切片：组装 prompt（资料片段 + 问题）→ `ChatClient` 生成回答，附带原文出处
4. 若资料不足（检索为空或相关度低）：调用通义 `enable_search` 联网作答，回答附带网页来源
5. 返回答案 + 来源列表：`sources:[{type, title, snippet, url?, materialId?, page?}]`，其中 `type=material|web`

### 4.3 AI 出题

1. 用户选一份资料 / 主题 → `QuizController` → `QuizService`
2. 取该资料的代表性切片作为上下文
3. 用结构化 prompt 让 LLM 输出 JSON 格式题目（选择 / 填空 / 简答）
4. 返回题目列表

## 5. 多模态提取器架构（可插拔）

不同格式的资料先经过对应的"提取器"统一转成纯文本，之后的切片 / 向量化 / 检索 / 问答完全复用同一套流程。新增一种格式 = 新增一个提取器，不改动核心。

```
上传文件 → 按类型分发：
   ├─ txt/PDF/Word → Apache Tika 提取文本
   ├─ 图片(jpg/png) → 通义 qwen-vl 视觉识别
   └─ 录音(mp3/wav) → 通义 paraformer 语音识别(加分项)
        ↓
   统一得到文本 → 切片 → 向量化 → 入库（后续流程一致）
```

### 接口设计

定义统一的提取器接口，例如：

```java
public interface ContentExtractor {
    boolean supports(String contentType);   // 是否支持该文件类型
    String extract(Resource file);          // 提取为纯文本
}
```

- `TikaExtractor` — 文本/PDF/Word
- `ImageExtractor` — 图片，调用 `qwen-vl`
- `AudioExtractor`（加分项）— 录音，调用 `paraformer`

`MaterialService` 持有一组 `ContentExtractor`，按文件类型选择匹配的一个执行。

### 范围说明

- **MVP**：文本类 + 图片
- **加分项**：录音转写（文件大、识别异步耗时、费用高，核心闭环稳定后再做）

## 6. 四人分工

| 成员 | 模块 | 主要工作 | 主要交付 |
|---|---|---|---|
| **1** | 用户与资料管理（后端） | Spring Security + JWT 登录注册、文件上传、MySQL、materials 增删查 | `AuthController/Service`、`MaterialController`、users/materials 表 |
| **2** | RAG 核心 + 提取器（后端） | 提取器(Tika/图片qwen-vl，录音为加分项) → 切片 → embedding → 入向量库；检索；`ChatClient` 问答；联网兜底 | `ContentExtractor` 系列、`MaterialService`、`RagService`、`ChatController` |
| **3** | 出题 + Prompt 工程（后端） | `QuizService`、prompt 模板、LLM 输出结构化 JSON 题目；（加分项：资料总结） | `QuizController/Service`、prompt 模板 |
| **4** | 前端（Vue3 + Element Plus） | 登录页、资料上传/列表页、问答页(显示出处)、出题/答题页 | 前端工程 |

接口先用 Apidog/Postman 约定接口契约，前后端并行开发。

## 7. 技术栈

- **后端框架**：Spring Boot 3.4.x
- **AI 框架**：spring-ai-alibaba 1.0.0.2（通过 BOM 管版本）
- **模型**：
  - `qwen-plus`（对话，开 `enable_search` 联网）
  - `text-embedding-v3`（向量）
  - `qwen-vl`（图片视觉识别）
  - `paraformer`（语音识别，加分项）
- **数据库**：MySQL
- **文档解析**：Apache Tika
- **鉴权**：JWT（jjwt）
- **向量库**：SimpleVectorStore（文件版，零部署）；有余力可换 PGVector / Redis
- **前端**：Vue3 + Element Plus + axios

## 8. 团队协作与交接

核心原则：**先一起定契约，再各自用 mock 解耦并行，最后按固定交接点对接。** 最大风险是互相卡住（前端等后端、出题等 RAG、所有人等登录）。

### 依赖关系

```
成员1 (登录/上传/DB) ──┬──> 成员2 (RAG核心) ──> 成员3 (出题，复用2的检索)
                       └──> 成员4 (前端，调所有人的接口)
```

- 成员1 是地基（2/3/4 都依赖），**最先出活**
- 成员2 是核心，成员3 复用其检索
- 成员4 调所有人接口，最怕等后端 → 用 mock 先行

### Day 1 全员一起定的"契约"

1. **项目骨架**：建好 Spring Boot 工程、包结构、推 git（成员1 主导）
2. **接口契约**（用 Apidog/Postman 定下路径/入参/返回 JSON）：
   - `POST /api/auth/login` → `{token}`
   - `POST /api/materials` (file) → `{materialId, filename, type, status}`
   - `GET /api/materials` → `[{id, filename, type, status}]`
   - `POST /api/chat` `{question}` → `{answer, sources:[{type, title, snippet, url?, materialId?, page?}]}`
   - `POST /api/quiz` `{materialId, type, count}` → `{questions:[...]}`
3. **数据格式**：题目 JSON、资料状态枚举、出处 sources 的结构写死，前后端共用

### 各自并行（用 mock 解耦）

| 成员 | 立刻能做、不等别人 | 解耦方式 |
|---|---|---|
| 1 | 登录注册、上传、建表（纯自有，最先跑通） | 无依赖 |
| 2 | 提取/切片/向量/检索/问答 | 先用本地写死的文件喂数据，再接 1 的上传 |
| 3 | 出题 prompt 调试 | 先 hardcode 一段上下文，再接 2 的检索 |
| 4 | 全部页面 | axios 先指向契约里的 mock JSON |

口诀：**先用"假输入"把自己那段跑通，再换成"真接口"。**

### 固定交接点

- **A（约第 3 天）**：成员1 登录+上传可用 → 成员2 把"读本地文件"换成"读上传文件"；成员4 接真实登录
- **B（约第 7 天）**：成员2 的检索方法稳定 → 成员3 把 hardcode 上下文换成调用检索；成员4 接真实问答
- **C（约第 10 天）**：全部真实接口就绪 → 成员4 替换所有 mock，全员联调

交接本质：**被依赖方先暴露稳定的方法/接口签名，依赖方先按签名编程，实现好后直接替换。**

### Git 协作

- `main` 保持可运行，不直接在上面写
- 每人开分支：`feat/auth`、`feat/rag`、`feat/quiz`、`feat/frontend`
- 按模块分包/分目录，尽量不碰同一文件；前端放独立目录 `frontend/` 与后端物理隔离
- 每天至少合并一次到 main，小步提交
