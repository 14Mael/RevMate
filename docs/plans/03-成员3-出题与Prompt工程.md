# 成员 3 — AI 出题与 Prompt 工程（后端）

> 任务级计划，不含代码。你负责出题，复用成员2 的检索。

**目标**：根据资料/主题自动生成结构化练习题（单选/填空/简答），LLM 输出稳定可解析的 JSON。

**依赖**：Plan 0（骨架、契约）。成员2 的 `retrieve(userId, query, topK)`（交接点 B 前可用 hardcode 上下文解耦）。

**对外提供**：`POST /api/quiz` 出题能力。

---

## 任务 1：题目数据结构与契约对齐

- 按 Plan 0 契约定好题目 DTO：`{stem, options?, answer, analysis}`，单选有 options，填空/简答无 options
- 定好出题请求：`{materialId, type, count}`
- **验收**：DTO 编译通过、与契约一致

## 任务 2：Prompt 模板（核心工作）

- 设计三套 prompt：单选 / 填空 / 简答，明确要求"严格输出 JSON 数组、字段固定、不要多余文字"
- 先用一段 **hardcode 的复习文本** 调通（不等成员2）
- 处理 LLM 偶尔输出多余内容：用 Spring AI 的结构化输出（BeanOutputConverter）或自己做 JSON 清洗
- **验收**：给定一段文本，能稳定返回可解析的题目 JSON，三种题型都行

## 任务 3：接入检索生成上下文

- 交接点 B 后：用成员2 的 `retrieve` 取该 materialId 的代表性切片作为出题上下文（或取该资料全部切片做采样）
- 把上下文喂给 prompt 出题
- **验收**：基于真实上传的资料能出题，题目内容确实来自资料

## 任务 4：出题接口

- 实现 `POST /api/quiz`：参数校验（count 上限、type 合法）→ 取上下文 → 调 LLM → 解析 → 返回题目列表
- **验收**：Postman 调用返回 count 道指定题型的题目

## 加分项（有余力再做）

- 资料总结：`POST /api/summary` 对一份资料生成摘要/知识点提纲

---

## 你的里程碑

- 第 1~7 天：任务 1~2（hardcode 文本调通 prompt，不被阻塞）
- 第 8~10 天：**交接点 B 后**任务 3~4（接真实检索）
- 第 11~ ：调题目质量、做加分项

## 自测

- 单元测试：JSON 解析/清洗、参数校验
- 手测：同一段资料反复出题，检查格式稳定性和题目合理性
