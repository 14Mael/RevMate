# RevMate 小组课堂展示分工说明

本文档用于课堂展示时说明 5 位成员分别讲什么、展示哪些代码和页面、每个人的工作重点是什么。建议展示时不要只说“我负责某某模块”，而是按“模块作用 -> 核心流程 -> 代码位置 -> 演示效果”的顺序讲，这样老师更容易看出每个人的实际贡献。

## 一、展示总思路

RevMate 是一个面向复习场景的智能学习助手，整体闭环是：

```text
用户登录 -> 创建课程/上传资料 -> 文本提取与切片 -> RAG 问答
       -> AI 出题 -> 自动/手动收录错题 -> 错题复习
       -> 推荐课程 -> 收藏学习资源
```

项目采用前后端分离架构：

| 层次 | 技术与职责 | 主要代码位置 |
| --- | --- | --- |
| 前端 | Vue 3 + Vite + TypeScript + Element Plus，负责页面、交互、路由、接口调用 | `frontend/src` |
| 后端 | Spring Boot + Spring Security + Spring Data JPA，负责接口、业务逻辑、鉴权和数据持久化 | `backend/src/main/java/com/team/study` |
| AI 能力 | Spring AI OpenAI 兼容接口，负责问答、出题、关键词提取、课程推荐整理 | `backend/src/main/java/com/team/study/service` |
| 数据库 | MySQL，保存用户、课程、资料、资料切片、聊天历史、收藏课程、错题 | `backend/src/main/resources/schema.sql` |
| 测试 | JUnit 5 + Mockito + Spring Boot Test，覆盖服务层、控制器、配置、提取器等 | `backend/src/test/java` |

推荐展示顺序：

1. 刘：项目总体架构 + RAG/向量检索核心。
2. 宋：前端整体工程结构、路由、布局和接口封装。
3. 文：智能问答、推荐课程两个功能。
4. 杨：AI 出题、错题本两个功能。
5. 王：测试覆盖、测试用例和质量保障。

如果你们想严格按“1 -> 2 -> 3 -> 4 -> 5”的顺序讲，也可以把宋的“前端整体结构”放到文、杨讲完四个功能之后，作为对前端实现方式的补充说明。

## 二、1 号刘：项目总体架构与 RAG 向量模块

### 1. 要讲的核心内容

刘主要负责把项目整体讲清楚，重点是后端架构和 RAG 问答流程。可以先讲 RevMate 为什么需要 RAG：普通 AI 只能靠通用知识回答，而本项目希望 AI 优先基于用户上传的复习资料回答，并返回资料来源。

可以按这条链路讲：

```text
资料上传
  -> 文件类型识别
  -> 文本提取
  -> 文本切片
  -> 可选 embedding 向量化
  -> material_chunks 入库
  -> 用户提问
  -> 当前课程/资料范围内检索相关切片
  -> 拼接上下文给大模型
  -> 返回答案 + 来源
```

### 2. 代码对应关系

| 讲解点 | 代码位置 | 说明 |
| --- | --- | --- |
| 总体入口 | `backend/src/main/java/com/team/study/RevMateApplication.java` | Spring Boot 启动类 |
| 数据表设计 | `backend/src/main/resources/schema.sql` | 用户、课程、资料、切片、聊天历史、错题等表 |
| 资料上传 | `MaterialController`、`MaterialServiceImpl` | 接收文件、校验用户和课程、保存资料元信息 |
| 文件处理 | `FileProcessingService` | 异步执行提取、预览转换、入库 |
| 提取器路由 | `ExtractorRouter`、`ContentExtractor` | 按 txt/pdf/word/image/audio 等类型选择提取器 |
| 文档提取 | `TikaExtractor`、`ImageExtractor`、`AudioExtractor` | 文本/PDF/Office、图片 OCR、音频转写 |
| 切片与检索 | `DocumentIngestionServiceImpl` | 资料切片、embedding 写入、关键词/向量混合检索 |
| 向量处理 | `SpringAiEmbeddingService`、`EmbeddingVectorCodec` | 调用 embedding 模型、序列化向量、余弦相似度 |
| RAG 回答 | `RagServiceImpl`、`ChatController` | 检索资料、构造 prompt、普通/流式问答 |

### 3. 建议展示的重点

刘可以重点讲以下 4 点：

1. **分层架构**：Controller 只负责接收请求，Service 处理业务逻辑，Repository 访问数据库，DTO 负责接口数据传输。
2. **用户隔离**：资料、切片、问答、错题都带 `userId`，检索时只在当前用户、当前课程范围内找资料。
3. **RAG 检索机制**：`material_chunks` 表保存切片文本和 embedding，提问时使用关键词分数和向量相似度进行召回。
4. **流式问答**：`POST /api/chat/stream` 用 SSE 返回 `delta` 和 `done` 事件，前端可以逐字显示答案。

### 4. 可直接使用的讲稿

> 我负责的是项目总体架构和 RAG 向量检索模块。我们项目采用前后端分离，后端用 Spring Boot 分成 Controller、Service、Repository、Entity 和 DTO 几层。用户上传资料后，系统会先保存资料，再根据文件类型调用不同的提取器，把 PDF、Word、图片或音频统一转成文本。之后通过 `DocumentIngestionServiceImpl` 对文本进行切片，并在开启 embedding 的情况下生成向量，存入 `material_chunks` 表。
>
> 当用户提问时，`RagServiceImpl` 会先在当前用户、当前课程或指定资料范围内检索相关切片。如果检索结果足够可靠，就把这些资料片段作为上下文交给大模型回答，并返回来源卡片；如果资料不足，则回退为通用回答。这样可以保证回答尽量基于用户自己的复习资料。

## 三、2 号文：智能问答与推荐课程

2 号建议讲两个功能：**智能问答**和**推荐课程**。这两个功能都属于用户直接看到的 AI 辅助学习能力。

### 3.1 智能问答

#### 要讲的功能

智能问答是项目首页，也是截图中展示的页面。它支持：

- 选择课程后提问。
- 基于上传资料进行 RAG 回答。
- SSE 流式输出，回答逐步显示。
- 展示参考来源，点击来源可以跳到对应资料。
- 保存聊天历史，新建对话和查看历史记录。
- 支持从资料预览页带引用文本回来提问。

#### 代码对应关系

| 讲解点 | 代码位置 | 说明 |
| --- | --- | --- |
| 问答页面 | `frontend/src/views/HomeView.vue` | 消息列表、输入框、来源卡片、历史面板 |
| 流式请求 | `frontend/src/api/chat.ts` | `chatStream` 使用 fetch 读取 SSE |
| 后端接口 | `backend/src/main/java/com/team/study/controller/ChatController.java` | `/api/chat` 和 `/api/chat/stream` |
| RAG 逻辑 | `backend/src/main/java/com/team/study/service/RagServiceImpl.java` | 检索资料、生成回答、返回来源 |
| 聊天历史 | `ChatHistoryController`、`ChatHistoryServiceImpl`、`ChatHistoryPanel.vue` | 保存、加载、删除历史会话 |

#### 建议展示方式

演示时可以现场问一个和资料相关的问题，例如“总结一下页面置换算法”，然后指出：

1. 页面右侧/下方会流式出现回答。
2. 回答不是凭空生成，而是来自当前课程资料。
3. 下方参考来源来自后端返回的 `sources`。
4. 聊天历史会保存，切换页面后还能恢复。

#### 可直接使用的讲稿

> 我讲的是智能问答功能。这个页面对应 `HomeView.vue`，用户选择课程后输入问题，前端会调用 `chatStream`，通过 SSE 接收后端一点点返回的内容，所以页面上能看到逐步生成的效果。后端对应的是 `ChatController` 和 `RagServiceImpl`，它会先检索当前用户当前课程下的资料切片，再把相关内容作为上下文交给大模型回答。回答完成后还会返回来源列表，前端把这些来源展示成参考卡片，用户可以看到答案依据来自哪份资料。

### 3.2 推荐课程

#### 要讲的功能

推荐课程用于在复习资料之外，为用户找到进一步学习的资源。它支持：

- 用户输入学习主题，系统联网搜索真实课程/视频资源。
- 后端对搜索结果去重、过滤低质量文档站、优先推荐 B 站、MOOC、技术社区等平台。
- AI 根据真实搜索结果整理课程标题、平台、链接、推荐理由和难度。
- 用户可以收藏课程，也可以查看和删除已收藏课程。

#### 代码对应关系

| 讲解点 | 代码位置 | 说明 |
| --- | --- | --- |
| 推荐课程页面 | `frontend/src/views/RecommendView.vue` | 搜索、结果列表、收藏课程 |
| 课程卡片 | `frontend/src/components/CourseCard.vue` | 统一展示课程标题、平台、理由、难度 |
| 前端 API | `frontend/src/api/courses.ts` | 推荐、收藏、删除收藏 |
| 后端接口 | `backend/src/main/java/com/team/study/controller/CourseController.java` | `/api/courses/keywords`、`/recommend`、`/saved` |
| 推荐逻辑 | `backend/src/main/java/com/team/study/service/CourseRecommendServiceImpl.java` | 搜索、重排、AI 整理、过滤虚假链接 |
| 收藏数据 | `saved_courses` 表、`SavedCourse` | 保存用户收藏课程 |

#### 建议展示方式

可以输入“操作系统 页面置换算法”或“Java 并发”，展示课程推荐列表，再点“收藏”，切到“已收藏”标签证明数据已保存。

#### 可直接使用的讲稿

> 推荐课程功能的目标是帮助用户在复习资料之外找到可继续学习的资源。前端页面是 `RecommendView.vue`，用户输入主题后会调用后端 `/api/courses/recommend`。后端不是直接让 AI 编造课程，而是先调用联网搜索服务获取真实搜索结果，再在 `CourseRecommendServiceImpl` 中进行去重、过滤和平台优先级排序，最后让 AI 基于这些真实结果整理成课程卡片。为了防止模型生成不存在的链接，后端还会校验推荐链接必须来自搜索结果。用户觉得有用的课程可以收藏到 `saved_courses` 表。

## 四、3 号杨：AI 出题与错题本

3 号建议讲两个功能：**AI 出题**和**错题本**。这两个功能共同构成“练习 -> 批改 -> 收录错题 -> 再练习”的复习闭环。

### 4.1 AI 出题

#### 要讲的功能

AI 出题支持：

- 选择课程。
- 可选指定某一份资料，也可以基于整门课程资料出题。
- 选择题型：单选题、填空题、简答题。
- 设置题目数量。
- 前端进入答题阶段，逐题作答。
- 提交后展示得分、正确答案和解析。
- 答错的单选题和填空题会自动加入错题本，答对的题也可以手动加入。

#### 代码对应关系

| 讲解点 | 代码位置 | 说明 |
| --- | --- | --- |
| 出题页面 | `frontend/src/views/QuizView.vue` | 配置、答题、回顾三个阶段 |
| 前端 API | `frontend/src/api/quiz.ts` | 调用生成题目接口 |
| 后端接口 | `backend/src/main/java/com/team/study/controller/QuizController.java` | `POST /api/quiz` |
| 出题逻辑 | `backend/src/main/java/com/team/study/service/QuizServiceImpl.java` | 根据资料上下文生成题目 |
| 题目 DTO | `QuizRequest`、`QuizResponse`、`QuestionItem` | 题目请求和响应结构 |
| 错题联动 | `frontend/src/api/wrongQuestion.ts`、`WrongQuestionServiceImpl` | 自动/手动收录错题 |

#### 建议展示方式

选择一个课程，题型选“单选题 + 填空题”，生成 3 道题。故意答错一道，提交后展示：

1. 回顾页显示正确答案和解析。
2. 错题自动加入错题本。
3. 可以跳到错题本页面看到这道错题。

#### 可直接使用的讲稿

> AI 出题功能对应 `QuizView.vue` 和后端 `QuizServiceImpl`。用户可以选择课程、资料、题型和题目数量。后端会从 `DocumentIngestionService` 取出对应资料或课程的文本切片作为上下文，然后根据题型选择不同的 Prompt 模板，比如单选题要求 4 个选项、填空题要求用空格占位、简答题要求输出参考答案和解析。模型输出后，后端会清洗并解析 JSON，返回统一的题目结构。前端分为配置、答题和回顾三个阶段，提交后会自动批改，并把答错的单选和填空题收录到错题本。

### 4.2 错题本

#### 要讲的功能

错题本用于沉淀用户薄弱点，支持：

- 查看所有错题。
- 按课程和题型筛选。
- 展示题干、选项、错误答案、正确答案、解析和错误次数。
- 标记“已掌握”。
- 删除错题。
- 重做错题。
- 基于错题生成“举一反三”练习题。

#### 代码对应关系

| 讲解点 | 代码位置 | 说明 |
| --- | --- | --- |
| 错题本页面 | `frontend/src/views/WrongBookView.vue` | 错题列表、筛选、重做、举一反三 |
| 答题组件 | `frontend/src/components/QuizRunner.vue` | 错题重做复用的答题组件 |
| 前端 API | `frontend/src/api/wrongQuestion.ts` | 保存、列表、删除、标记掌握、强化练习 |
| 后端接口 | `backend/src/main/java/com/team/study/controller/WrongQuestionController.java` | 错题本 REST API |
| 错题逻辑 | `backend/src/main/java/com/team/study/service/WrongQuestionServiceImpl.java` | 保存去重、次数统计、强化练习 |
| 数据表 | `wrong_questions` 表 | 保存题干、答案、解析、错误次数、掌握状态 |

#### 建议展示方式

展示错题本列表时可以强调：

1. 同一用户同一课程下，重复题干不会产生多条，而是更新错误次数。
2. 错题支持标记掌握，掌握后不再进入“待复习”。
3. “举一反三”是基于原错题生成同知识点新题。

#### 可直接使用的讲稿

> 错题本功能承接 AI 出题模块。用户答错题后，前端会调用错题本接口批量保存错题。后端 `WrongQuestionServiceImpl` 会根据用户、课程和题干哈希进行去重，如果同一道题再次答错，不会重复新增，而是增加 `wrongCount` 并更新最近错误时间。错题本页面支持按课程和题型筛选，也可以标记掌握、删除或重做错题。另外我们还做了“举一反三”功能，后端会把原题、答案和解析组成 Prompt，让 AI 生成同知识点的新练习题，帮助用户针对薄弱点继续练习。

## 五、4 号宋：前端整体实现

4 号建议不要再重复介绍四个业务功能，而是讲“这些功能在前端是怎么组织起来的”。重点是前端工程结构、导航布局、路由守卫、API 封装、状态管理和交互体验。

### 1. 要讲的核心内容

宋可以讲：

- 前端使用 Vue 3 + Vite + TypeScript。
- 页面通过 Vue Router 组织，登录后进入主布局。
- `AppLayout.vue` 实现顶部导航、用户信息、退出登录和页面缓存。
- `src/api` 对后端接口进行统一封装。
- `http.ts` 统一处理 token、错误提示、401 跳转登录。
- `Pinia` 保存登录用户和 token。
- 业务页面按功能拆成 `views`，公共组件放在 `components`。
- 样式变量统一放在 `assets/styles`，保证整体视觉一致。

### 2. 代码对应关系

| 讲解点 | 代码位置 | 说明 |
| --- | --- | --- |
| 前端入口 | `frontend/src/main.ts`、`frontend/src/App.vue` | 创建 Vue 应用 |
| 主布局 | `frontend/src/layouts/AppLayout.vue` | 顶部导航、用户区、页面容器、keep-alive |
| 路由 | `frontend/src/router/index.ts` | 登录页、主页、资料、推荐、错题本、出题 |
| 登录状态 | `frontend/src/stores/user.ts` | token 和用户名持久化 |
| HTTP 封装 | `frontend/src/api/http.ts` | baseURL、Bearer token、统一错误处理 |
| 类型定义 | `frontend/src/api/types.ts` | 前后端接口数据结构 |
| 页面目录 | `frontend/src/views` | Login、Home、Materials、Recommend、Quiz、WrongBook |
| 公共组件 | `frontend/src/components` | 课程卡片、历史面板、答题组件等 |
| 样式系统 | `frontend/src/assets/styles` | 全局样式、变量、Element Plus 覆盖 |

### 3. 建议展示的重点

可以打开 `AppLayout.vue` 和 `router/index.ts` 讲：

1. 导航栏包含“智能问答、我的资料、推荐课程、错题本、AI 出题”五个入口。
2. `router.beforeEach` 做登录守卫，未登录不能访问业务页面。
3. `keep-alive` 缓存业务页，切换菜单后保留问答或答题状态。
4. `http.ts` 自动加 `Authorization: Bearer token`，不用每个页面重复写鉴权逻辑。

### 4. 可直接使用的讲稿

> 我负责前端整体实现。前端使用 Vue 3、Vite 和 TypeScript，页面结构通过 Vue Router 管理。登录后进入 `AppLayout.vue`，顶部导航包含智能问答、我的资料、推荐课程、错题本和 AI 出题几个模块。我们还使用 `keep-alive` 缓存主要页面，比如用户在 AI 出题页面答到一半，切换到其他页面后再回来，状态不会立即丢失。
>
> 接口调用统一封装在 `src/api` 下，其中 `http.ts` 会自动给请求加上 JWT token，并统一处理 401、403、服务器错误和业务错误。这样业务页面只需要调用对应 API 函数，不需要关心重复的请求细节。整体上，前端把页面、组件、接口和类型分开管理，使后续维护更清晰。

## 六、5 号王：测试与质量保障

### 1. 要讲的核心内容

王主要讲项目如何保证功能可靠。可以从测试覆盖面、重点测试用例、运行方式三个角度讲。

本项目后端测试主要覆盖：

- Controller 接口层：聊天、课程推荐、错题本等接口是否调用正确服务。
- Service 业务层：RAG、出题、推荐课程、错题本、资料管理、学科管理、聊天历史等核心逻辑。
- Extractor 提取器：Tika、音频转写、提取器路由。
- Security 安全：JWT 生成和校验。
- Config 配置：CORS、OpenAI 兼容配置、Jackson 读取限制等。
- 工具类：embedding 向量序列化、反序列化、余弦相似度。

### 2. 代码对应关系

| 测试方向 | 测试文件 | 说明 |
| --- | --- | --- |
| RAG 问答 | `RagServiceImplTest`、`RagRetrievalCalibrationTest` | 分数阈值、资料命中、流式事件、通用回答回退 |

### 3. 推荐重点讲的测试样例

王可以挑 3 个最有代表性的测试讲：

1. **RAG 检索可靠性测试**
   - `RagServiceImplTest` 校验向量分数低时不会错误使用资料回答。
   - 防止问“第 6 章”却混入“第 3 章”的低分切片。

2. **推荐课程防幻觉测试**
   - `CourseRecommendServiceImplTest` 校验 AI 返回的链接必须来自真实搜索结果。
   - 如果模型输出坏 JSON，会回退使用搜索结果生成课程卡片。

3. **错题本去重测试**
   - `WrongQuestionServiceImplTest` 校验同一用户、同一课程、同一题干重复错题不会产生多条记录。
   - 重复答错会更新 `wrongCount`。

### 5. 可直接使用的讲稿

> 我负责测试和质量保障。我们项目的测试主要集中在后端，使用 JUnit 5、Mockito 和 Spring Boot Test。测试范围覆盖了 Controller、Service、提取器、安全配置、外部搜索服务和工具类。比如 RAG 模块中，我们测试了向量分数和关键词分数的阈值，确保低相关资料不会被错误引用；课程推荐模块中，我们测试了 AI 不能返回搜索结果之外的链接，防止模型编造课程；错题本模块中，我们测试了重复错题会合并统计，而不是重复插入。
